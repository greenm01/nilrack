# Dreyer's English Style Guide for nilrack Docs

This guide enforces a crisp, punchy, and human tone for all project documentation. It is designed to eradicate "AI smell" and corporate filler.

## The Rules of the Red Pen

1. **Purge the Weasel Words:** Never use *very*, *rather*, *quite*, *somewhat*, or *actually* unless they serve a rhythmic purpose.
2. **Kill the Corporate Slop:** Words like *leveraging*, *robust*, *dynamic*, *seamless*, *comprehensive*, and *canonical* are banned. If a plugin host cannot react to change, it is broken; don't brag about it.
3. **Shorten the Throat-Clearing:** Do not start sentences with "In order to," "The fact that," or "It is important to note that." Just say the thing.
4. **Active Verbs Only:** nilrack *hosts* plugins; it doesn't "leverage an implementation of" plugin hosting.
5. **Vary the Rhythm:** AI loves long, balanced sentences. Humans use fragments. And short sentences. It keeps the reader awake.
6. **Narrative Over Lists:** Don't default to bulleted lists for every explanation. Use them for technical specs, but use paragraphs to explain *why* something exists.
7. **Be Opinionated:** Documentation should sound like it was written by an engineer with a point of view, not a committee. Admit when something is a trade-off.

## Prohibited Phrases & Replacements
- "Leveraging X" -> "Using X" / "Built on X"
- "A robust solution for" -> "A tool for"
- "Seamlessly integrates" -> "Works with"
- "In order to" -> "To"
- "Provides the ability to" -> "Can" / "Does"
- "Dynamic" -> (Delete it)
